# SVR Feeds (Smart Value Recapture)

- [https://docs.chain.link/data-feeds/svr-feeds](https://docs.chain.link/data-feeds/svr-feeds) — Explains Chainlink Smart Value Recapture (SVR) Feeds: an extension of Chainlink Price Feeds that recaptures oracle-extractable value (OEV), mainly liquidation-related non-toxic MEV, via an optional private transmission route and auction. Describes MEV vs OEV, SVR’s dual-aggregator architecture (standard public route + private route), MEV-Share/Atlas auction flow, and fail-safe fallback to standard feed after a configurable delay if the private route fails. Integration is minimal: protocols read SVR feeds exactly like standard feeds using AggregatorV3Interface but must use an SVR-enabled feed address and complete a compatibility form. Covers variants (general vs Aave-only), supported networks, revenue split concept, and risks (delay, liquidation competition, MEV not eliminated, dynamic recapture rates).. Key details: SVR extends standard Chainlink Price Feeds with an optional private transmission flow; same aggregator interface/data structure; Dual transmission: (1a) Standard Aggregator via public mempool; (1b) SVR Aggregator via private channel (e.g., Flashbots MEV-Share); Auction: searchers bid to backrun liquidation after oracle update; builder selects highest bid and bundles liquidation in same block; Fail-safe: if private route fails/times out, SVR feed reverts to Standard Feed price after a configurable delay (seconds); Integration steps: (1) fill out required form for compatibility; (2) find SVR feed address on Feed Addresses page (filter SVR); (3) read via AggregatorV3Interface using SVR address; Supported auction systems by network: Ethereum Mainnet uses Flashbots MEV-Share; Base/Arbitrum/BNB Chain use Atlas; SVR is targeted at non-toxic liquidation-related OEV; not for frontrunning/sandwich attacks; Economics: recaptured OEV revenue is split between integrating protocol and Chainlink Network; split may change; Developer responsibilities/ToS apply as with Chainlink Data Feeds; protocols must configure risk parameters appropriately
- [https://docs.chain.link/data-feeds/svr-feeds/searcher-onboarding-atlas](https://docs.chain.link/data-feeds/svr-feeds/searcher-onboarding-atlas) — Explains how existing Chainlink SVR searchers integrate with the Atlas auction system on Base, Arbitrum, and BNB Chain to participate in non-Ethereum-mainnet SVR auctions. Covers core components (SVR price feeds, Atlas contracts/offchain infra, solver network), supported protocols (Aave, Compound, Venus), operational bonding recommendations, and contract/feed reference addresses (Atlas v1.6.4, DappControl, and per-asset aggregator addresses). Describes the auction flow: receive user operations over a JSON-RPC WebSocket, simulate solver operations (failed sims never hit chain), sign required payloads, and submit solutions. Key caveats: listen to aggregator (not proxy) addresses; final bundled tx is submitted by Chainlink oracles; solvers must bond native tokens for gas reimbursement; must sign an exact oracle-provided gasPrice; solver gas limit must not exceed DappControl.getSolverGasLimit; configure WebSocket buffers >10KB and implement auto-reconnect.. Key details: Onboarding contact: [email protected]; Atlas concept: MEV-Share-like; oracle update + liquidation represented as EIP-712 messages bundled atomically (ERC-4337/multicall-like) without block builders; Final bundled transaction is submitted by Chainlink oracle network, not the searcher; Bond native tokens with Atlas contracts to reimburse oracle gas for solver op + oracle update; recommended bonding: Base/Arbitrum 0.1 ETH, BNB Chain 1 BNB; Gas price chosen by Chainlink oracle and provided at auction start; solver must sign gasPrice exactly equal to provided value; Do not exceed SolverGasLimit; query via DappControl.getSolverGasLimit; higher gas limit SolverOps will fail; WebSocket payloads may exceed default buffers; set read/write buffers > 10KB; implement auto-reconnect due to disconnects; Endpoints: bid https://svr-bid-endpoint.chain.link/ ; WebSocket wss://svr-bid-endpoint.chain.link/ws/solver; Signing format (EIP-191 message): "<auctionID>:<userOperationHash>:<solverOperationFrom>" (colon-delimited); ethers.js Signer.signMessage referenced; Feed caveat: listen to Aggregator address for price reports (not proxy); get aggregator by reading proxy.aggregator(); use Aave-SVR feeds for Aave and SVR feeds for other protocols
- [https://docs.chain.link/data-feeds/svr-feeds/searcher-onboarding-ethereum](https://docs.chain.link/data-feeds/svr-feeds/searcher-onboarding-ethereum) — Explains how to onboard and operate as a searcher for Chainlink Smart Value Recapture (SVR) liquidation opportunities on Ethereum Mainnet using Flashbots MEV-Share. Covers the end-to-end flow: monitoring private MEV-Share events for SVR price update transactions, identifying forwarder calls, decoding nested calldata to extract the updated feed address and median price from transmitSecondary reports, then constructing and bidding a liquidation bundle to backrun the price update in the same block (e.g., Aave liquidations). Includes TS and Go patterns for connecting to the MEV-Share stream, filtering by the forward() function selector, handling single-tx vs bundle events, and decoding ABI-encoded report bytes. Notes that updates come from multiple forwarder contracts (per Node Operator Proxy) and that bundles are nonce-ordered.. Key details: Onboarding contact: [email protected]; SVR price updates are delivered via private mempool (Flashbots Protect / MEV-Share), not public mempool; Filter MEV-Share pending tx events by forward(address to, bytes callData) selector 0x6fadcf72; Forwarder contracts vary by Node Operator Proxy; updates may originate from different forwarder addresses; forward(callData) contains an encoded transmitSecondary(bytes32[3] reportContext, bytes report, bytes32[] rs, bytes32[] ss, bytes32 rawVs) call to the SVR aggregator; Report bytes decode to: uint32 observationsTimestamp, bytes32 observers, int192[] observations, int192 juelsPerFeeCoin; updated price taken as median observation; MEV-Share stream can emit single-transaction events or bundle events; bundle txs are in ascending nonce order; Searcher submits liquidation tx in same bundle as price update; highest bid wins and MEV is shared with protocol (e.g., Aave) and Chainlink Network

# MVR Feeds (Multiple-Variable Response)

- [https://docs.chain.link/data-feeds/mvr-feeds](https://docs.chain.link/data-feeds/mvr-feeds) — Explains Chainlink Multiple-Variable Response (MVR) feeds, which bundle multiple related data points (including numeric and non-numeric types) into a single onchain update, unlike traditional single-value feeds. Describes benefits (multi-field bundles, extensibility without redeploying contracts, single-transaction efficiency, flexible types). Shows how to read MVR data via the BundleAggregatorProxy interface: fetch the proxy address from the SmartData addresses list (filter via “Show Multiple-Variable Response (MVR) feeds”), call latestBundle() to get a bytes payload, use bundleDecimals() for per-numeric-field decimals, and latestBundleTimestamp() for last update time, then decode bytes into the documented structure. Caveat: MVR feeds only store the latest bundle onchain; historical data must be captured by your contract or an offchain indexer. Links to Solidity, ethers.js v5, and Viem (TS) implementation guides and API reference.. Key details: MVR interface contract: BundleAggregatorProxy; latestBundle() returns the full bundle as bytes; bundleDecimals() returns decimal places for each numeric field; latestBundleTimestamp() returns last update timestamp; Integration steps: get proxy address -> call latestBundle() -> decode bytes into expected struct; Find addresses/data structures on SmartData Addresses page; filter with “Show Multiple-Variable Response (MVR) feeds” checkbox; MVR feeds store only the most recent onchain bundle; historical requires own storage or offchain indexer; Example bundle fields can include uint256 values and bool (e.g., openToNewInvestors)

- [https://docs.chain.link/data-feeds/mvr-feeds/guides/ethersjs](https://docs.chain.link/data-feeds/mvr-feeds/guides/ethersjs) — Guide for consuming Chainlink Multiple-Variable Response (MVR) feeds in JavaScript using ethers.js v5. Explains that MVR feeds store multiple values in a single onchain bytes bundle and you must obtain the correct BundleAggregatorProxy address plus the exact bundle field types/decimals from the SmartData Addresses page (“MVR Bundle Info”). Shows how to define a minimal ABI, create a JsonRpcProvider and Contract, validate freshness via latestBundleTimestamp() with a heartbeat-aligned staleness threshold, fetch latestBundle() and decode with ethers.utils.defaultAbiCoder.decode(dataStructure, bundleBytes), then apply per-field decimals from bundleDecimals() (skipping bools) and format with ethers.utils.formatUnits. Includes .env setup and a complete reusable client example. Warns example code is unaudited and not production-ready; ethers v6 APIs differ.. Key details: SDK/language: JavaScript (Node.js) with ethers@^5.x; note ethers v6 API differences; Minimal ABI: latestBundle() returns (bytes), bundleDecimals() returns (uint8[]), latestBundleTimestamp() returns (uint256); Provider setup: new ethers.providers.JsonRpcProvider(process.env.RPC_URL); Contract setup: new ethers.Contract(proxyAddress, bundleAggregatorProxyABI, provider); .env variables: RPC_URL and MVR_FEED_ADDRESS; recommend adding .env to .gitignore; Staleness check: compare now (Date.now()/1000) vs latestBundleTimestamp(); threshold should be based on feed heartbeat + buffer (not arbitrary); Decoding: ethers.utils.defaultAbiCoder.decode(dataStructure, bundleBytes); dataStructure MUST exactly match feed’s documented field types; Field mapping: use parallel fieldNames[] to build an object from decodedValues[]; Decimal handling: call bundleDecimals(); handle provider return-type variability; for numeric fields compute divisor = BigNumber.from(10).pow(decimalPlaces); store raw, adjusted, decimals, formatted; Formatting helper: ethers.utils.formatUnits(value, decimals) for human-readable strings; skip bool fields for decimal adjustment
- [https://docs.chain.link/data-feeds/mvr-feeds/guides/evm-solidity](https://docs.chain.link/data-feeds/mvr-feeds/guides/evm-solidity) — Explains how to consume Chainlink Multiple-Variable Response (MVR) Data Feeds on EVM chains using Solidity. MVR feeds return multiple fields packed into a single onchain bytes bundle; consumers must (1) find the feed’s BundleAggregatorProxy address and exact bundle struct layout/decimals from the SmartData Addresses page, (2) call latestBundle() to fetch bytes, (3) validate freshness via latestBundleTimestamp() against a staleness threshold derived from the feed heartbeat, (4) decode with abi.decode(bytes,(StructType)) using an identically ordered/types struct, and (5) apply per-field scaling using bundleDecimals() (divide by 10**decimals[i]) while noting Solidity uint256 division truncates. Includes guidance on custom errors, array-length checks for decimals, and storing raw vs scaled values. Disclaimer: example code is unaudited; do not use in production without audits/best practices.. Key details: Find BundleAggregatorProxy address + exact bundle field types/order/decimals in SmartData Addresses -> "MVR Bundle Info"; Interface import: @chainlink/contracts/src/v0.8/data-feeds/interfaces/IBundleAggregatorProxy.sol; IBundleAggregatorProxy functions used: latestBundle() returns bytes, bundleDecimals() returns uint8[], latestBundleTimestamp() returns uint256; Struct used for decoding must match feed layout exactly (same order and Solidity types) or decoding will be wrong; Staleness check pattern: (block.timestamp - lastUpdateTime) <= STALENESS_THRESHOLD; threshold should align to feed heartbeat + buffer (not arbitrary); Decode pattern: Data memory decoded = abi.decode(rawBundle, (Data)); Decimals scaling: actual = raw / (10 ** decimals[i]); non-numeric fields typically have decimals 0; Caveat: uint256 division truncates; keep raw values or scale offchain if precision is required; Safety checks: custom errors (e.g., StaleData, InsufficientDecimals) and validate decimals array length before indexing
- [https://docs.chain.link/data-feeds/mvr-feeds/guides/viem](https://docs.chain.link/data-feeds/mvr-feeds/guides/viem) — Guide for consuming Chainlink Multiple-Variable Response (MVR) Data Feeds in TypeScript using the viem library. Explains that MVR feeds store multiple values packed into a single onchain bytes bundle and must be decoded with an exact, feed-specific schema from the SmartData Addresses page (“MVR Bundle Info”). Walks through defining the BundleAggregatorProxy ABI (latestBundle, bundleDecimals, latestBundleTimestamp), creating a viem PublicClient and reading from a proxy address, building type-safe TS structures, checking staleness via timestamp vs a threshold aligned to the feed heartbeat, decoding bytes with decodeAbiParameters using a matching types array, and applying per-field decimals using bundleDecimals + formatUnits. Includes a complete reusable MVRFeedClient class. Warns example code is unaudited and not production-ready without additional checks.. Key details: SDK/language: TypeScript using viem; Contract methods used: latestBundle() -> bytes, bundleDecimals() -> uint8[], latestBundleTimestamp() -> uint256; Define ABI via viem parseAbiItem([...]) and use client.readContract({ address, abi, functionName }); Environment variables: RPC_URL and MVR_FEED_ADDRESS loaded via dotenv; proxyAddress typed as `0x${string}`; Staleness check: compare latestBundleTimestamp to current unix time; default threshold 86400s; should be based on feed heartbeat + buffer (not arbitrary); Decoding: decodeAbiParameters(parameterStructure, bundleBytes) where parameterStructure is derived from a feed-specific dataTypes array that MUST exactly match the feed schema; Mapping decoded tuple to named fields via fieldNames array; raw values are bigint/boolean; Decimal handling: fetch bundleDecimals array and format bigint values with formatUnits(value, decimals)

# Operations and Infrastructure

- [https://docs.chain.link/data-feeds/contract-registry](https://docs.chain.link/data-feeds/contract-registry) — Explains Chainlink’s Flags Contract Registry: an onchain source of truth for which Chainlink price feed proxy contracts are officially owned/operated by Chainlink and currently active on supported networks. A proxy address that returns true when queried against the network’s Flags contract indicates authenticity and active status; inactive feeds are removed and the registry is actively maintained as new feeds deploy. Shows how to query the registry onchain in Solidity by calling getFlag(address) on the IFlags interface, wiring the correct network-specific Flags contract address in the consumer contract constructor, and exposing a helper like isOfficialChainlinkFeed(proxy). Includes a table of Flags contract addresses per chain (e.g., Ethereum, Arbitrum, Optimism, Polygon, zkSync, etc.).. Key details: Registry purpose: verify price feed proxy contracts are Chainlink-owned/operated and active; A getFlag(proxy) == true implies (1) official Chainlink feed and (2) currently active; Inactive feeds are removed from the registry; list is actively maintained; Solidity interface: interface IFlags { function getFlag(address) external view returns (bool); }; Consumer pattern: store IFlags flags; initialize with network Flags contract address in constructor(address flagsContractAddress); Helper function example: isOfficialChainlinkFeed(address proxy) public view returns (bool) { return flags.getFlag(proxy); }; Requires using the correct Flags contract address for the target network; Provides a table of Flags contract addresses for supported chains (Arbitrum, Avalanche, Base, BNB Chain, Celo, Ethereum, Gnosis, Linea, Mantle, Optimism, Polygon, Scroll, Sonic, UniChain, zkSync, etc.)
- [https://docs.chain.link/data-feeds/data-sources](https://docs.chain.link/data-feeds/data-sources) — Explains how Chainlink Data Feeds and Data Streams source underlying data across multiple independent aggregators, vendors, and authoritative entities. Provides a table mapping asset/feed types to their sourcing models and minimum vendor/aggregator counts (e.g., most market feeds use 3+ vendors; some long-tail crypto uses a single source). Highlights special sourcing for SmartData Proof of Reserve (PoR) and NAV feeds (directly from custodians/fund administrators/auditors/asset managers or on-chain data for cross-chain PoR), and notes delayed data for UK/Euro ETFs (15-minute delay). Points to a separate page for evaluating data quality, liquidity/distribution risks, single-source provider risks, and mitigation strategies; suggests contacting Chainlink for more details.. Key details: Crypto Price feeds: sourced from 3+ crypto price aggregators and/or market data vendors using CEX and/or DEX data; Crypto State Price feeds: 3+ vendors using DEX pool state data; Forex/Precious Metals/US Oil/US Equities: typically 3+ market data vendors; UK & Euro ETFs: 2+ market data vendors with 15-minute delayed data; SmartData PoR (offchain): direct from custodian/fund administrator/auditor/security auditor/asset manager/regulated appraiser/regulator-approved source; SmartData PoR (cross chain): on-chain data; SmartData NAV (offchain): direct from Fund Administrator or Asset Manager; Exchange Rate Feeds: on-chain data; Composite Index Feeds: DON-calculated index values from 3+ aggregators/vendors; Data Link Long Tail Crypto: single source from 1 crypto price data aggregator
- [https://docs.chain.link/data-feeds/deprecating-feeds](https://docs.chain.link/data-feeds/deprecating-feeds) — Explains Chainlink’s process and rationale for deprecating certain Data Feeds due to low usage or lack of economic sustainability, as part of a shift toward Chainlink Economics 2.0 and optimizing blockspace/operator costs. Informs users that feeds may be shut down, relocated, or replaced over time and that data quality monitoring stops two weeks before the deprecation date. Advises developers to be careful selecting feeds: official docs feeds are reviewed, but community/custom deployments may carry additional risks and require due diligence. Provides a network-by-network table of feeds scheduled for deprecation with shutdown dates and key feed parameters (pair, deviation threshold, heartbeat, decimals, and onchain aggregator address). Points users to Chainlink Discord (data-feeds-user-notifications) and a contact link for updates/questions.. Key details: Deprecation candidates: low/no usage or no viable path to economic sustainability; Data quality monitoring ceases 2 weeks prior to the deprecation date; Status updates: join official Chainlink Discord and subscribe to data-feeds-user-notifications; Caveat: feeds listed in official documentation are reviewed; community/custom deployments may have additional risks; Deprecation table fields: Network, Pair, Deprecation date, Deviation, Heartbeat (seconds), Decimals, Aggregator address, Asset name/type, Market hours; Examples of listed networks include Arbitrum, Base, BNB Chain, BOB, Ethereum, Katana, Monad, Ronin, Sonic, TRON
- [https://docs.chain.link/data-feeds/developer-responsibilities](https://docs.chain.link/data-feeds/developer-responsibilities) — Explains developer responsibilities when integrating Chainlink Data Feeds, emphasizing that feed performance can be affected by external market conditions and by the integrator’s own application code. Defines two risk classes: Market Integrity Risks (e.g., manipulation and market risk, especially low-liquidity assets) and Application Code Risks (code quality, audits, dependencies). States developers are solely responsible for monitoring and mitigating these risks, informing users, and securely managing interactions with third-party services. Recommends implementing risk mitigation processes such as data quality checks, circuit breakers, and contingency logic. Points to related guides for Data Feed Categories and Evaluating Data Source Risks for further assessment and mitigation techniques.. Key details: Developers are solely responsible for monitoring and mitigating market integrity risks affecting feed behavior; Market manipulation examples listed: spoofing, ramping, bear raids, cross-market manipulation, wash trading, frontrunning; Low-liquidity/high market-risk assets are highlighted as more vulnerable to manipulation; Developers should understand asset market risks before integrating a feed and inform end users of applicable risks; Developers must institute mitigation processes: data quality checks, circuit breakers, and contingency logic; Application code responsibilities include ensuring code quality/reliability, auditing before production deployment, and vetting/auditing dependencies/imports; References additional docs: Data Feed Categories and Evaluating Data Source Risks
- [https://docs.chain.link/data-feeds/l2-sequencer-feeds](https://docs.chain.link/data-feeds/l2-sequencer-feeds) — Explains Chainlink L2 Sequencer Uptime Data Feeds for rollups where sequencer downtime can break normal L2 read/write access and create unfair advantages. Shows how to use the feed to detect downtime, pause sensitive actions (e.g., liquidations), and enforce a post-recovery grace period before trusting other data feeds. Lists supported networks and their sequencer uptime feed proxy addresses. Describes update/relay mechanics: Arbitrum uses OCR-triggered updates via ValidatorProxy/ArbitrumValidator and L1 inbox messaging to ArbitrumSequencerUptimeFeed; other networks (Base, OP, etc.) relay status from L1 through OptimismValidator and cross-domain messengers to an L2 uptime feed. Provides a Solidity consumer example using AggregatorV2V3Interface.latestRoundData(), interpreting answer (0 up / 1 down) and using startedAt to enforce GRACE_PERIOD_TIME. Notes example is unaudited and not for production.. Key details: Sequencer uptime feed read via AggregatorV2V3Interface.latestRoundData(); Interpretation: answer==0 => sequencer up; answer==1 => sequencer down; Use startedAt from uptime feed as timestamp when status last changed; enforce grace period: block.timestamp - startedAt > GRACE_PERIOD_TIME; Example grace period constant: GRACE_PERIOD_TIME = 3600 (seconds); Arbitrum updates: OCR round ~every 30s; nodes call ArbitrumValidator.validate via ValidatorProxy; message to Arbitrum inbox; L2 ArbitrumSequencerUptimeFeed.updateStatus sets status and records L1 timestamp; Other networks: L1 AggregatorProxy/Aggregator -> OptimismValidator.validate -> L1CrossDomainMessenger.sendMessage -> CanonicalTransactionChain.enqueue -> sequencer -> L2CrossDomainMessenger -> OptimismSequencerUptimeFeed.updateStatus(bool status, uint64 timestamp); Outage ordering guarantee (non-Arbitrum): messages enqueued in CanonicalTransactionChain are processed in arrival order after sequencer returns, so the 'sequencer down' flip executes before later dependent transactions; Supported network proxy addresses listed for: Arbitrum, Base, Celo, Mantle, MegaETH, Metis, OP, Scroll, Soneium, X Layer, zkSync
- [https://docs.chain.link/data-feeds/self-managed-feeds](https://docs.chain.link/data-feeds/self-managed-feeds) — Explains Chainlink Data Feeds “self-managed feeds,” where onchain values are written by a third party (the chain or its operator) rather than Chainlink node operators via Chainlink Labs–managed infrastructure. Describes the high-level architecture: a Chainlink Runtime Environment (CRE) workflow reads from Chainlink Data Streams and writes updates to a proxy contract so consumers can use standard Data Feeds interfaces. Clarifies responsibilities and lack of guarantees: Chainlink Labs does not manage the publication path, monitoring, heartbeat/deviation compliance, latency, correctness of onchain writes, or SLAs. Notes visibility constraints: self-managed feed proxy addresses are not published in Chainlink’s public feed catalogs; integrators must obtain addresses directly from the chain/operator.. Key details: Onchain updates are written by a third party (chain/operator), not Chainlink node operators; CRE workflow reads from Chainlink Data Streams and writes to a proxy contract; Consumers read via the familiar Chainlink Data Feeds proxy interface; Chainlink Labs provides no monitoring for heartbeat compliance, deviation compliance, end-to-end latency, or correctness of onchain writes; No SLAs for self-managed feed publication; Proxy addresses are not listed in official Chainlink public documentation; obtain from chain/operator; Contact for deploying self-managed feeds: [email protected]

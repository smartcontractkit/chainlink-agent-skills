# Using Data Feeds

- [https://docs.chain.link/data-feeds/historical-data](https://docs.chain.link/data-feeds/historical-data) — Explains how to retrieve historical Chainlink Data Feeds values using AggregatorV3Interface.getRoundData(roundId) and how “roundId” differs between proxy contracts and underlying aggregators. Describes update triggers (deviation threshold, heartbeat) and why history may span multiple aggregator implementations behind a proxy. Details proxy roundId encoding as (phaseId << 64) | aggregatorRoundId, where phaseId increments on aggregator upgrades and aggregatorRoundId starts at 1; proxy roundIds always increase though aggregator roundIds may be non-monotonic. Shows how to derive phaseId via right shift by 64 and aggregatorRoundId via uint64 mask, then loop through roundIds to fetch history and detect end-of-phase by revert. Warns that looping onchain can be very gas-expensive and that valid roundIds must be known; incomplete rounds may have zero timestamps.. Key details: Use AggregatorV3Interface.getRoundData(uint80 roundId) to fetch historical values; latestRoundData() provides current proxy roundId; Proxy roundId formula: uint80(uint256(_phase) << PHASE_OFFSET | _originalId) (conceptually (phaseId << 64) | aggregatorRoundId); phaseId increments when the proxy upgrades to a new underlying aggregator; aggregatorRoundId is the underlying aggregator’s roundId starting at 1; Compute phaseId offchain: phaseId = roundId >> 64; compute aggregatorRoundId: uint64(roundId) or (roundId & 0xFFFFFFFFFFFFFFFF); To loop current phase history: startRoundId = latestRoundId - aggregatorRoundId + 1; increment until latestRoundId; To query previous phases: set roundId = (phaseId << 64) | 1 and increment until getRoundData reverts (end of that aggregator’s history); getRoundData returns (roundId, answer, startedAt, updatedAt, answeredInRound); answeredInRound is deprecated; Caveat: roundIds are not guaranteed incremental/continuous; you must know a valid roundId; zero timestamps indicate incomplete rounds; Onchain looping can be extremely gas costly, especially if state changes in the same function
- [https://docs.chain.link/data-feeds/selecting-data-feeds](https://docs.chain.link/data-feeds/selecting-data-feeds) — Explains how to evaluate and select Chainlink Data Feeds based on “market pricing risk” and other data-quality considerations, emphasizing that developers are responsible for assessing accuracy, availability, and suitability. Defines feed categories (Low/Medium/High/Very High Market Pricing Risk, New Token, Custom, Deprecating) and describes typical risk drivers: liquidity/volume, venue concentration, cross-rate effects, migrations/market events, limited sources, hacks/bridge failures/delistings, and extreme volatility. Advises reading Risk Mitigation and Evaluating Data Sources sections, monitoring feeds, configuring protocol risk parameters appropriately, and noting that higher-risk feeds may be deprecated or wound down under the Data Feed Shutdown Policy. Custom Feeds require extra scrutiny and include onchain single-source, proof of reserve, exchange rate (not market price), technical metrics, TVL, and custom index feeds. Recommends joining Chainlink Discord and checking the Changelog for updates.. Key details: Feed categories: Low, Medium, High, Very High Market Pricing Risk; New Token; Custom; Deprecating; Low risk feeds: standardized workflow, multiple sources aggregated by node operators; still requires protocol risk parameter configuration; Medium risk drivers: low/inconsistent liquidity, venue price spreads, market concentration on one exchange, cross-rate risk, migrations/market events, provider spread, changing source availability; High risk: heightened Medium-risk factors; user acknowledges responsibility; may be deprecated—see Data Feed Shutdown Policy; Very High risk: extreme events (hack/bridge failure/delisting), project deprecation, extremely low volumes, very limited reliable sources; may be wound down; Chainlink may not separately monitor; New Token feeds: insufficient history for rigorous assessment; probationary period then reclassified (Very High/High/Medium/Low) or rarely deprecated; users must verify liquidity/stability; Custom Feeds are use-case-specific and may not suit general use; contact Chainlink Labs for details; Custom feed subtypes include: Onchain Single Source, Onchain Proof of Reserve, Exchange Rate (reads external onchain contract; not equivalent to market price feeds), Technical (e.g., Fast Gas/Block Difficulty), TVL, Custom Index
- [https://docs.chain.link/data-feeds/using-data-feeds](https://docs.chain.link/data-feeds/using-data-feeds) — Explains how to read Chainlink Data Feeds on any EVM-compatible chain, noting the request/response format is consistent across feed types and networks while addresses differ. Covers required inputs: RPC endpoint URL, LINK token contract address (per-network), and the specific feed (aggregator) contract address (per-network). Provides onchain consumer examples (Solidity, Vyper) using AggregatorV3Interface and latestRoundData(), and offchain read examples using web3.js, ethers.js, Web3.py, and go-ethereum. Highlights that latestRoundData returns five values and points to the API reference. Includes key caveats: examples use hardcoded values and unaudited code (not production-ready); on L2s you must check the L2 Sequencer Uptime Feed to ensure data accuracy during sequencer outages.. Key details: Variables needed: RPC endpoint URL, LINK token contract address (network-specific), feed/aggregator contract address (network-specific); Onchain Solidity imports AggregatorV3Interface from @chainlink/contracts/src/v0.8/shared/interfaces/AggregatorV3Interface.sol; Solidity consumer pattern: store AggregatorV3Interface dataFeed; set in constructor with hardcoded aggregator address; call dataFeed.latestRoundData() and return answer; latestRoundData() returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound); L2 requirement: consult L2 Sequencer Uptime Feed before trusting feed answers during sequencer outages; Offchain web3.js pattern: new Web3(RPC_URL); new web3.eth.Contract(ABI, addr); priceFeed.methods.latestRoundData().call(); Offchain ethers.js pattern: new ethers.providers.JsonRpcProvider(RPC_URL); new ethers.Contract(addr, ABI, provider); priceFeed.latestRoundData(); Offchain Python pattern: Web3.HTTPProvider(RPC_URL); web3.eth.contract(address=addr, abi=abi); contract.functions.latestRoundData().call(); Examples use Sepolia BTC/USD aggregator address 0x1b44F3514812d835EB1BDB0acB33d3fA3351Ee43 and RPC https://rpc.ankr.com/eth_sepolia; All examples are explicitly unaudited and use hardcoded values; not for production

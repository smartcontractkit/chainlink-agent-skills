# General

- [https://docs.chain.link/data-feeds](https://docs.chain.link/data-feeds) — Overview of Chainlink Data Feeds and how to integrate them to access onchain real-world data (asset prices, RWA reserves/NAV/AUM, rates/volatility, L2 sequencer health). Explains feed categories: Price Feeds, SmartData, Multiple-Variable Response (MVR) bundles, Smart Value Recapture (SVR/OEV recapture), Rate & Volatility, L2 Sequencer Uptime, and Self-managed feeds (not operated/monitored/listed by Chainlink Labs). Describes core architecture: consumer reads via interfaces from proxy contracts that point to upgradeable aggregator contracts; best practice is to read through the proxy to remain resilient to aggregator upgrades. Shows how to inspect proxy/aggregator configuration via ABI/block explorers (e.g., Etherscan), and notes aggregators can differ by network; use typeAndVersion() and check source/config. Mentions @chainlink/contracts package and links to address catalogs and API references.. Key details: EVM consumers typically read single-value feeds via AggregatorV3Interface against a proxy contract; Common proxy example: EACAggregatorProxy.sol; proxy points to an underlying aggregator that can be upgraded; MVR consumers use IBundleAggregatorProxy; proxy exposes latestBundle(), bundleDecimals(), latestBundleTimestamp(); Inspect onchain configuration via ABI or block explorers (e.g., Etherscan): aggregator address, owner, latestRoundData(), latestAnswer, latestTimestamp; Best practice: call latestRoundData() on the proxy (not directly on the aggregator) to avoid breakage during upgrades; Aggregators vary across networks; identify with typeAndVersion() and verify source/config (e.g., Arbitrum BTC/USD differs); Proxy and aggregator have owner addresses with permissions to change configuration; feeds may be updated for migrations/rebrands/market events; Self-managed feeds: updates written by chain/third-party (e.g., CRE workflow reading Data Streams); Chainlink Labs does not operate/monitor/list addresses; Integration aid: @chainlink/contracts NPM package; use feed address catalogs for networks/addresses

- [https://docs.chain.link/data-feeds/llms-full.txt](https://docs.chain.link/data-feeds/llms-full.txt) — API reference for Chainlink Data Feeds contracts. Explains best practice for reading feeds via the proxy using Solidity AggregatorV3Interface (instantiate with the feed’s proxy address) and optionally querying the underlying aggregator contract (AccessControlledOffchainAggregator) when proxy methods are insufficient. Documents core AggregatorV3Interface methods (decimals, description, getRoundData, latestRoundData, version) including full return tuples and notes that answeredInRound is deprecated. Describes AccessControlledOffchainAggregator as the behind-proxy aggregator; recommends using proxy to avoid aggregator upgrades breaking apps; suggests using typeAndVersion() to identify aggregator type/version and checking source/config because implementations vary by network/feed. Mentions using explorer ABIs (e.g., Etherscan/Arbiscan) and Remix AtAddress to inspect variables/functions. Notes minAnswer/maxAnswer are largely unused and apps should implement their own circuit breakers if needed; several legacy getter functions are deprecated.. Key details: Instantiate AggregatorV3Interface with the feed proxy address (example: Sepolia ETH/USD 0x694AA1769357215DE4FAC081bf1f309aDC325306); AggregatorV3Interface.decimals() external view returns (uint8); AggregatorV3Interface.description() external view returns (string memory); AggregatorV3Interface.getRoundData(uint80 _roundId) returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound); AggregatorV3Interface.latestRoundData() returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound); answeredInRound is marked deprecated (historical multi-round computation); AggregatorV3Interface.version() external view returns (uint256); AccessControlledOffchainAggregator: prefer calling proxy interface so aggregator changes don’t affect app; read aggregator directly only for non-proxy functions; Use typeAndVersion() on aggregator to identify aggregator type/version; aggregator implementations can differ across networks (e.g., BTC/USD on Arbitrum differs); Deprecated aggregator functions called out: getAnswer(), getTimestamp(), latestRound() (avoid); latestAnswer lacks timestamps for freshness checks
- [https://docs.chain.link/data-feeds/ens](https://docs.chain.link/data-feeds/ens) — Explains how to resolve Chainlink Data Feed contract addresses via ENS using the data.eth namespace. Shows naming convention for feeds (e.g., eth-usd.data.eth) and how subdomains return different related addresses (proxy, underlying aggregator, proposed aggregator). Describes architecture: a single Chainlink resolver per network (Ethereum Mainnet resolver address provided) that manages data.eth subdomains. Notes that aggregator/proposed addresses can change during upgrades; the resolver emits AddrChanged events keyed by the feed’s namehash, enabling listeners to detect changes. Reverse lookup is explicitly not supported. Provides JavaScript (Web3) and Solidity examples for resolving ENS names/bytes32 node hashes, including ENS registry address and guidance to use namehash/subgraph/npm package to compute hashes. Includes warnings that Solidity example is unaudited and not for production.. Key details: Feed ENS suffix: data.eth; format: <base>-<quote>.data.eth (e.g., eth-usd.data.eth, btc-usd.data.eth); Subdomain prefixes map to addresses: proxy.<name> (proxy), aggregator.<name> (underlying aggregator), proposed.<name> (proposed aggregator); Ethereum Mainnet Chainlink resolver (for data.eth): 0x122eb74f9d0F1a5ed587F43D120C1c2BbDb9360B; Resolver emits AddrChanged events when subdomain addresses change; filter using indexed parameter = namehash('<subdomain>.<feed>.data.eth') (example uses ethers.utils.namehash('aggregator.eth-usd.data.eth')); Reverse lookup is not supported; ENS registry address (mainnet/testnet): 0x00000000000C2E074eC69A0dFb2997BA6C7d2e1e; Solidity resolvers take bytes32 node (namehash) not string; hash IDs obtainable via subgraph or npm package eth-ens-namehash; Example ETH/USD namehash provided: 0xf599f4cd075a34b92169cf57271da65a7a936c35e3f31e854447fbb3e7eb736d; Solidity example includes explicit warning: unaudited, do not use in production
